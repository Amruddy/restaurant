# Feature Spec: Booking

## Цель

Дать гостю быстрый способ оставить заявку на бронирование стола или другого ресторанного формата.

Бронирование не должно обещать автоматическое подтверждение. На начальном этапе заявка должна попадать администратору, а администратор принимает решение и отвечает гостю в кратчайшее время.

## Роль в пользовательском пути

Бронирование относится к первому уровню важности вместе с атмосферой и меню.

Пользователь может перейти к бронированию:

- с первого экрана;
- после просмотра меню;
- из блока контактов;
- из блока семейных событий или банкетов;
- после знакомства с атмосферой ресторана.

## Модель бронирования

Тип бронирования: заявка с подтверждением администратором.

Сайт не должен писать, что стол уже забронирован сразу после отправки формы. Правильный смысл сообщения: заявка принята, администратор свяжется с гостем и подтвердит возможность бронирования.

## Данные заявки

Обязательные данные:

- имя;
- телефон;
- дата;
- время;
- количество гостей.

Дополнительные данные:

- комментарий.

Повод визита не является обязательным полем. Главное для администратора - понимать дату, время, контакт и количество гостей.

## Что можно бронировать

Бронирование должно быть достаточно универсальным, чтобы пользователь мог оставить заявку на:

- обычный стол;
- семейный формат;
- бизнес-встречу;
- банкет;
- другой ресторанный формат.

Если позже появятся отдельные правила для банкетов, семейных комнат или приватных залов, их нужно описать в отдельной спецификации событий или расширении этой спецификации.

## Отправка заявки

Заявка должна доходить до администратора.

Канал отправки для MVP: Telegram через серверную часть Next.js.

Конкретные технические детали отправки описываются в `docs/technical-spec.md`. В этой feature spec важно требование: заявка не должна теряться, а администратор должен получить достаточно данных, чтобы ответить гостю.

## Сообщение после отправки

После отправки пользователь должен получить понятное сообщение:

- заявка принята;
- администратор свяжется для подтверждения;
- бронь не считается подтвержденной автоматически.

Сообщение не должно обещать мгновенное подтверждение или гарантированную бронь.

## Исключительные случаи

- Если обязательные поля не заполнены, пользователь должен понять, что нужно исправить.
- Если заявка не отправилась, пользователь должен увидеть альтернативный способ связи.
- Если выбранное время невозможно подтвердить, это решает администратор после получения заявки.

## Ограничения

- Не делать автоматическое подтверждение брони без отдельной спецификации.
- Не обещать свободные столы в реальном времени без интеграции с системой ресторана.
- Не требовать повод визита как обязательное поле.
- Не описывать здесь визуальное оформление формы.
- Не описывать здесь техническую реализацию отправки.

## Открытые вопросы

- Нужен ли альтернативный номер телефона или мессенджер рядом с формой.
- Нужны ли отдельные правила для банкетов, семейных комнат или приватных форматов.

# Feature Spec: Booking

## Цель

Дать гостю быстрый способ оставить заявку на бронирование стола.

На текущем этапе сайт не подтверждает бронь автоматически. Гость отправляет заявку, а администратор связывается с ним и подтверждает возможность посадки.

## Роль в пользовательском пути

Бронирование относится к первому уровню важности вместе с атмосферой и меню.

Пользователь может перейти к бронированию:

- с первого экрана;
- после просмотра меню;
- из блока контактов;
- после знакомства с атмосферой ресторана.

На мобильных устройствах путь к бронированию должен быть доступен без долгого поиска, но не должен мешать первому экрану и корзине.

При переходе к бронированию форма должна быть компактной: пользователь должен сразу видеть основные поля без лишней высокой шапки блока.

## Модель бронирования

Тип бронирования: заявка с подтверждением администратора.

Пользователь должен:

1. Указать имя.
2. Указать телефон.
3. Выбрать дату.
4. Выбрать время.
5. Указать количество гостей.
6. При необходимости оставить комментарий.
7. Отправить заявку.
8. Получить сообщение, что администратор свяжется для подтверждения.

Сайт не должен писать, что стол уже забронирован сразу после отправки формы.

## Данные заявки

Обязательные данные:

- имя;
- телефон;
- дата;
- время;
- количество гостей.

Дополнительные данные:

- комментарий.

Отдельное поле для сценария посещения не нужно. Главное - дата, время, контакт и количество гостей.

## Что можно бронировать

На первом этапе форма бронирования покрывает обычную заявку на стол.

Специальные сценарии не входят в сценарий текущего MVP.

Если позже ресторан захочет добавить банкеты, приватные залы, семейные комнаты или автоматическую проверку доступности, нужно будет обновить спецификации отдельно.

## Уведомление администратора

После отправки заявки администратор должен получить уведомление.

Для MVP уведомление уходит в Telegram через серверную часть Next.js.

Уведомление должно содержать:

- имя;
- телефон;
- дату;
- время;
- количество гостей;
- комментарий;
- пометку, что это заявка, а не автоматически подтвержденная бронь.

Telegram-уведомление является каналом получения заявки, но не системой учета свободных столов.

## Сообщение после отправки

После отправки пользователь должен получить понятное сообщение:

- заявка принята;
- администратор свяжется для подтверждения;
- бронь не считается подтвержденной автоматически.

Если заявку не удалось отправить, пользователь должен понять причину или увидеть альтернативный способ связи.

## Ошибочные состояния

Нужно предусмотреть:

- не заполнены обязательные поля;
- телефон указан в некорректном формате;
- заявка не отправилась из-за технической ошибки;
- Telegram-переменные окружения не настроены на сервере.

Технические ошибки не должны показывать пользователю секреты, токены или служебные детали.

## Отмена и изменение брони

На первом этапе изменение или отмена брони происходит через контакт с рестораном.

Если позже нужна самостоятельная отмена или изменение брони на сайте, это нужно описать отдельно.

## Ограничения

- Не обещать автоматическую бронь.
- Не показывать доступные слоты без системы проверки доступности.
- Не считать заявку подтвержденной бронью.
- Не добавлять специальные сценарии в текущую форму бронирования.
- Не хранить персональные данные в `localStorage`.
- Не описывать здесь визуальное оформление формы.

## Открытые вопросы

- Нужен ли альтернативный номер телефона или мессенджер рядом с формой.
- Какие реальные часы работы использовать в форме.
- Нужны ли отдельные правила для банкетов, семейных комнат или приватных форматов.
- Нужна ли в будущем автоматическая проверка свободных столов и сохранение броней в базе.

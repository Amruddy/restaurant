# Technical Spec

## Цель

Описать техническую основу MVP сайта ресторана до начала разработки.

Этот документ не описывает визуальный стиль и пользовательские тексты. Он фиксирует стек, структуру приложения, хранение данных, клиентские сценарии, отправку заявок и технические ограничения.

## Стек

Проект строится на Next.js.

Основной стек:

- Next.js;
- React;
- TypeScript;
- CSS Modules или глобальные CSS-файлы, если этого достаточно;
- локальные ассеты в `public/`;
- структурированные данные меню в отдельном файле данных.

Next.js выбран потому что MVP включает не только статическую страницу, но и интерактивные сценарии:

- меню с категориями;
- полное меню и популярные позиции;
- корзина доставки;
- заявка на бронирование;
- формы доставки;
- будущая онлайн-оплата;
- API routes или server actions для бронирования, доставки и уведомлений.

## Принципы архитектуры

- Не усложнять архитектуру до появления реальной необходимости.
- Держать пользовательские сценарии отдельно: меню, бронь, доставка, контакты.
- Не хранить секреты в клиентском коде.
- Не подключать платежи без отдельной спецификации онлайн-оплаты.
- Не добавлять CMS или админку в MVP без отдельного решения.
- Сохранять мобильный сценарий как основной.

## Предлагаемая структура

Ориентировочная структура после перехода на Next.js:

- `app/` - страницы, layout и маршруты Next.js.
- `components/` - переиспользуемые UI-компоненты.
- `features/` - функциональные области: menu, booking, delivery, contacts.
- `data/` - структурированные данные меню и временные данные MVP.
- `public/` - изображения, иконки, орнаменты и другие ассеты.
- `docs/` - спецификации проекта.

Финальная структура может быть уточнена перед кодом, но она должна поддерживать разделение функций и не превращать все приложение в один большой компонент.

## Страницы и маршруты

Для MVP можно начать с одной основной страницы, но структура должна позволять расширение.

Минимальные маршруты:

- `/` - главная страница с атмосферой, меню, бронированием, доставкой и контактами.
- `/menu/[id]` - страница отдельного блюда с фото, описанием, ценой и деталями.

Возможные маршруты позже:

- `/menu` - полное меню, если главная станет перегруженной.
- `/delivery` - отдельный сценарий доставки и корзины.

Решение о дополнительных страницах принимается после проверки прототипа и контента.

## Данные меню

Меню должно храниться структурированно.

Для MVP рекомендуется файл данных:

- `data/menu.ts`.

Выбор: используем `menu.ts`, потому что проект строится на Next.js и TypeScript, а меню будет использоваться сразу в нескольких сценариях: просмотр меню, популярные позиции, доставка и корзина.

Каждая позиция меню должна поддерживать данные из `docs/feature-specs/menu.md`:

- название;
- категория;
- описание;
- цена;
- фото;
- доступность для доставки;
- признак популярной или основной позиции.

Если позже появится CMS или админка, структура данных должна быть достаточно понятной, чтобы ее можно было перенести.

## Корзина доставки

Корзина работает на стороне клиента.

Минимальные требования:

- добавление позиции;
- изменение количества;
- удаление позиции;
- расчет суммы блюд;
- проверка минимальной суммы 500 рублей;
- расчет доставки 200 рублей для заказов от 500 до 999 рублей;
- бесплатная доставка от 1000 рублей;
- итоговая сумма с доставкой.

Корзина должна сохраняться в `localStorage`, чтобы пользователь не терял заказ при перезагрузке страницы.

При этом `localStorage` используется только для пользовательской корзины. Нельзя хранить там платежные секреты, персональные данные дольше необходимого или служебные ключи.

## Бронирование

Бронирование работает как заявка с подтверждением администратора.

Пользователь заполняет форму и получает сообщение, что администратор свяжется для подтверждения. Сайт не должен обещать автоматически подтвержденную бронь.

Минимальная техническая модель:

- клиентская форма бронирования;
- серверный обработчик Next.js для приема заявки;
- валидация обязательных полей на клиенте и сервере;
- отправка уведомления администратору в Telegram;
- успешное и ошибочное состояние для пользователя.

Заявка должна содержать:

- имя;
- телефон;
- дату;
- время;
- количество гостей;
- комментарий.

Токен Telegram-бота и идентификатор чата должны храниться в переменных окружения, а не в коде.

На текущем этапе не нужна база данных, учет столов и проверка доступных слотов. Если позже понадобится автоматическое бронирование, это нужно описать отдельным расширением спецификации.

## Доставка

Заказ доставки должен отправлять администратору:

- состав корзины;
- сумму блюд;
- стоимость доставки;
- итоговую сумму;
- имя;
- телефон;
- адрес;
- комментарий.

Техническое решение для MVP: заказ доставки отправляется администратору в Telegram через серверную часть Next.js.

Сообщение в Telegram должно содержать достаточно данных, чтобы администратор мог обработать заказ.

На первом этапе не делаем админ-панель. Если позже понадобится история заказов доставки, статусы или кабинет администратора, это нужно описать отдельной спецификацией.

## Онлайн-оплата

Онлайн-оплата входит в целевой MVP и описана в `docs/feature-specs/payment.md`.

До реализации нужно уточнить:

- момент создания заказа;
- обработку успешной оплаты;
- обработку ошибки оплаты;
- правила отмены и возврата;
- способ уведомления администратора;
- требования к безопасности и хранению данных.

Выбранный платежный провайдер: Сбер. СБП должен поддерживаться как способ оплаты, если он доступен в подключенном платежном сценарии.

Принятый сценарий MVP: платежная страница Сбера с доступными способами оплаты, включая СБП при наличии поддержки. Тестовые доступы обязательны. Webhook/callback обязателен для подтверждения оплаты. Возвраты на первом этапе обрабатываются вручную администратором через кабинет Сбера.

Нельзя реализовывать онлайн-оплату только на клиенте и нельзя хранить платежные секреты в репозитории.

## Контакты и карта

Контакты должны использовать реальные данные ресторана.

Карта добавляется только при наличии точного адреса и выбранного способа интеграции.

До выбора карты нельзя подключать случайные embed-виджеты или внешние скрипты без необходимости.

## Ассеты

Все основные изображения и декоративные материалы должны лежать в `public/`.

Ассеты должны соответствовать `docs/visual-rules.md`.

Не использовать внешние изображения как постоянную зависимость готового MVP, если их можно хранить локально.

## SEO и метаданные

MVP должен предусмотреть базовые SEO-данные:

- title;
- description;
- Open Graph title;
- Open Graph description;
- Open Graph image;
- корректный язык страницы;
- базовую структуру заголовков.

Подробная SEO-разметка может быть отдельной задачей, но базовые метаданные нужны в MVP.

## Деплой

Проект должен быть готов к деплою на Vercel.

Vercel выбран как основная платформа деплоя для Next.js-проекта.

## Ограничения

- Не добавлять CMS или админ-панель без отдельной причины.
- Не хранить секреты в репозитории.
- Не подключать онлайн-оплату без отдельного spec.
- Не использовать внешние CDN без необходимости.
- Не делать desktop-first реализацию без проверки мобильного сценария.
- Не начинать кодить функции, которые не описаны в feature specs.
- Не обещать автоматическую бронь без отдельной системы проверки доступности и хранения броней.

## Открытые вопросы

- Какие технические детали интеграции со Сбером и СБП будут подтверждены после получения тестовых доступов.
- Нужна ли в будущем автоматическая проверка свободных столов и какой провайдер базы использовать для этого.
- Можно ли объединять столы для компаний больше 6 гостей, если появится автоматическая бронь.
